# 越南信贷 SQL Skill V2（主路由）

## 核心职责

主 Skill 只负责以下内容：

1. 场景推荐与用户确认
2. 子 Skill 路由与组合加载
3. 跨场景关联目录
4. 公共口径（枚举语义、JSON 字段目录、SQL 约束）

主 Skill 不承载全量表结构。具体表字段与域内 join 只在子 Skill 维护。

## 每次分析必须执行的流程

1. 根据用户问题先推荐场景组合。
2. 明确要求用户确认最终场景列表。
3. 若用户选择多个场景，强制要求指定主场景（锚点场景）。
4. 仅加载已确认场景对应的参考文档。
5. 固定追问用户输出模式：`你是需要：1) 执行 SQL 看结果；2) 只要 SQL 代码？`; 默认走 `只输出 SQL`。
6. 若用户选择 `1) 执行 SQL 看结果`，加载 `script/sql_execution_python.py` 并按该脚本执行。
7. 若用户选择 `2) 只要 SQL 代码` 或未明确答复，禁止加载执行脚本，仅输出 SQL。
8. 若场景未确认或多场景未指定主场景，停止 SQL 生成并追问。

## 场景路由

- 用户分析 -> `references/user-domain.md`
- 申请分析 -> `references/application-decision-domain.md`
- 支用分析 -> `references/loan-domain.md`
- 额度分析 -> `references/credit-limit-domain.md`
- 风险分析 -> 推荐组合后由用户确认
- SQL 执行（仅在用户选择执行结果时） -> `script/sql_execution_python.py`

风险分析推荐组合规则：

- 默认：申请与决策域 + 额度域
- 包含 `KYC` / `画像` / `认证`：追加用户域
- 包含 `放款后` / `逾期` / `回收` / `账龄`：追加借据域

## 跨场景关联目录（统一口径）

本节只定义跨域关联规范，不重复子 Skill 的全量字段。

### 1) 用户域 <-> 申请与决策域

- 推荐 key：`borrower_no`
- 关联方向：用户 -> 申请（常见）
- 推荐锚点：用户分析用用户域；申请分析用申请域
- 风险与去重：1 个用户对应多次申请；按指标选择 `DISTINCT borrower_no` 或 `DISTINCT application_no`

### 2) 申请与决策域 <-> 借据域

- 推荐 key：`credit_no`（优先）/ `borrower_no` / `application_no` 与 `loan_apply_no`（需业务确认映射）
- 关联方向：申请 -> 借据（授信后支用链路）
- 推荐锚点：申请分析以申请域为锚；支用分析以借据域为锚
- 风险与去重：1 次申请可能映射 0~N 笔借据；先在锚点侧聚合再 join

### 3) 申请与决策域 <-> 额度域

- 推荐 key：`credit_no`（主）+ `borrower_no`（补充）
- 关联方向：申请决策结果 -> 额度结果
- 推荐锚点：申请分析以申请域；额度分析以额度域
- 风险与去重：单用户多额度/多调整记录；先明确额度快照口径（时点/最新/历史）

### 4) 借据域 <-> 额度域

- 推荐 key：`credit_no`
- 关联方向：额度 -> 借据占用/使用
- 推荐锚点：贷后表现以借据域；额度利用率以额度域
- 风险与去重：单额度可关联多笔借据；先按 `credit_no` 或 `loan_no` 聚合

### 5) 用户域 <-> 额度域

- 推荐 key：`borrower_no`
- 关联方向：用户 -> 额度
- 推荐锚点：用户经营分析以用户域；额度经营分析以额度域
- 风险与去重：用户多额度、多调整流水；先锁定用户层或额度层指标口径

## 公共字段口径

- `Adm`：`1=通过`，`0=拒绝`
- `kyc_status`：
  - `0` 未开始/验证码登录
  - `1` 基础信息
  - `2` 工作信息
  - `3` 联系人
  - `4` 身份证
  - `5` 活体
  - `6` 绑卡/完件
- `deleted`：`0=未删除`，`1=已删除`

## JSON / 半结构化字段总目录

以下字段可能为 JSON（或 JSON 字符串），使用前先确认结构并明确解析键。

### 用户域

- `t_trace_record.ext_params`：埋点扩展参数（JSON 文本）

### 申请与决策域

- `t_credit_application.risk_data`：申请侧风控输入特征
- `t_decision_result.result_data`：决策结果明细（常见键：`Adm`、`YN_creditamt`、`biz_action_type`、`YN_reject_code`）
- `yn_apply_feature_table.risk_data`：贷前特征主存储字段（重点，JSON）

### 借据域

- `t_loan_application.risk_data`：借款申请风控输入
- `t_loan_application.risk_check_result`：风控检查结果（常见含 `Adm`、`YN_reject_code`）
- `t_loan_contract.contract_vars`：合同变量（模板填充数据）
- `t_loan_product_snapshot.full_config`：产品配置快照


### 业务相关注意事项
- `总述`：产品为类7XX产品，即期限为7天的产品，砍头息XX%,即假设贷款1000，到手1000 * (1-XX%)的金额，且从贷款当天为第1天，还会收取利息。且该产品为单笔非循环贷产品，因此额度并不会恢复，因此一个授信号只会对应一个借据号(当前业务下,credit_no和loan_no一一对应)

- `t_credit_application`：所有授信申请都会在该表中
  - `首借`：borrower_no第一次有created_at可以认为是客户的首次借款，且product_code='7DAY_FAST_LOAN'
  - `开API`：borrower_no新增的product_code且对该客户来说该新增product_code为第一次出现,产品名抽象为'7DAY_FAST_LOAN_0X'(单客户上限6个，即最大06)
  - `提额`：该客户只要有借据结清，就会触发提额策略,且提额针对的是当前借据对应的产品(也可以被称为商户)

- `t_decision_result.result_data`：
  - `biz_action_type`：业务场景
    - `01` 首次授信(P1产品授信)
    - `02` P1产品提额-L2场景(L2,第一次提额)
    - `03` P1产品提额-L3/L3+场景(L3/L3+,第二/N次提额)
    - `04` 产品为开API申请
    - `05` 除P1产品提额


## SQL 统一约束

- 只生成 `MySQL` SQL。
- 默认只输出 SQL，不输出解释。
- 不自动猜测粒度、时间字段、去重 key、过滤条件。
- 缺分析对象、主事实表、指标定义、统计粒度、时间字段、去重口径、过滤条件时必须追问。
- 禁止口径猜测后直接构建代码：任何业务口径（如年龄计算时点、收入口径、区域口径、去重口径、时间边界）必须先与用户明确确认，再进行 SQL/代码构建与执行。
- 不默认排除 `deleted=1`，是否排除必须确认。
- 金额默认保留原始单位，是否转元必须确认。
- 尽量单条 SQL；默认不使用 CTE，除非用户明确要求；仍只输出 SQL。
- 采用“最短路径实现”：优先在主事实表完成核心指标；仅当主表无法表达指标时再引入 join/子查询。
- 对“首笔/首借/首次出现”等样本定义，必须先在主事实表完成样本判定（如 `ROW_NUMBER()` + `rn=1`），再关联结果表取决策/金额字段。
- 每个 join 必须有必要性；能单表完成的不引入跨表关联。
- 若关联键在业务上已确认为唯一键，禁止再叠加 `MAX(id)`、额外窗口或重复去重子查询制造冗余复杂度。
- 采用“最小变更原则”：用户未明确要求的口径、字段、过滤、分组、排序、衍生指标一律不新增。
- 用户已确认主口径后，禁止擅自并列替代口径或扩展为多版本输出。
- 当用户指定实现方式（如窗口函数、子查询形态）时，优先遵循该实现方式，不擅自改写为其他套路。
- 业务定义与实现形式分离：可调整写法风格，但不得改变已确认指标定义。

## 比率与去重防错规则（通用）

- 当用户需求包含“计数 + 比率/占比”时，优先给最小可用 SQL，不主动扩展为多口径、多层 CTE、明细+汇总并行等复杂版本。
- 比率类指标必须满足“同粒度分子 / 同粒度分母”；分子、分母与统计对象的去重粒度保持一致。
- 存在 `1:N` join 风险时，禁止在 join 后直接计算比率；必须先在目标统计粒度完成去重或聚合，再计算比率。
- 若比率依赖明细侧字段，先将明细侧收敛到目标粒度（去重/聚合）后再参与比率计算。
- 同一指标若存在多个可选业务口径，默认只输出用户确认的主口径；未明确要求时不并列输出多口径版本。
- 安全冗余按需开启：仅在用户要求稳健性或场景确有歧义/重复风险时增加防御性写法；用户要求简洁时保持最简实现。
- 输出前必须自检三项：`统计对象粒度`、`去重 key`、`分子分母一致性`。任一不明确先追问，再生成 SQL。

## 逾期与历史口径规则（通用）

- 当前状态不等于历史表现：凡“截至当前是否曾发生”类指标，不得仅用当前 `status` 直接替代历史/累计口径。
- 若主表已有稳定沉淀口径字段（如最大逾期天数、累计标识），优先直接使用；避免重复构造同义口径。
- “FPD/逾期”类指标必须先确认是“当前逾期状态”还是“历史最大逾期表现”，未确认必须追问。
- “截至当前/至今”类指标必须显式锚定统计时点（如 `CURDATE()` 或用户指定日期），不得隐式假设。

## SQL 写作风格（默认约束）

- 口径正确性 > 风格一致性。若风格与业务口径冲突，可偏离风格，但不得偏离口径。
- 先锁定指标定义、去重 key、时间字段、统计粒度，再输出 SQL。
- 在不影响口径正确性的前提下，先确认并遵循用户 SQL 风格偏好（如 CTE/非 CTE、简洁关联/分层写法）。
- 默认禁止使用 `WITH` / CTE 写法；仅当用户明确要求使用 CTE 时才允许使用。
- 优先单条 SQL；需要分层时优先使用内联子查询/普通关联，不使用 CTE；避免无意义嵌套。
- join 必须显式写出关联 key；存在 1:N 风险时先在明细侧聚合后再 join。
- 风格对齐优先：优先遵循用户明确的写作偏好（是否使用 CTE、是否使用别名、是否前缀列名、是否展开/嵌套）。
- 无 join 或无歧义时，避免引入不必要别名和列前缀；仅在消除歧义时使用。
- 若 `SELECT` 为多行字段列表，使用 leading comma 风格（逗号置于下一行行首），并且 `,` 后不加空格（示例：`,col_b`）。
- 首次类口径优先窗口函数模板：`ROW_NUMBER() OVER (PARTITION BY 业务键 ORDER BY 时间字段)` + `rn = 1`。
- 若用户要求“首次/首笔/首次出现”，必须保留首次判定条件；只能简化写法，不能简化口径。
- 表别名与字段别名保持清晰、一致、可读，避免歧义缩写。
- 统计字段使用语义化命名，建议使用 `numerator_*`、`denominator_*`、`*_rate`。
- 率类指标必须在 SQL 中显式体现分子和分母口径。
- 默认只输出 SQL，不输出解释性文字。

## 输出前自检清单（通用）

- 是否严格只实现了用户明确提出的需求范围（无未授权扩展）。
- 是否遵循了用户指定的主口径与实现方式。
- 是否保证了指标粒度、去重口径、分子分母的一致性。
- 是否匹配了用户的 SQL 风格偏好（简洁/稳健、CTE/非 CTE、别名使用）。
- 是否把“活跃/出现过”误当成“首次”口径。
- 交付 SQL 是否为单一确认版本（未授权不输出并行多版本）。
- 是否误把“当前状态”当作“历史表现”使用。
- 是否在主表可直接计算时仍引入了不必要 join。
- 对“首次类”需求，是否先在主事实表完成样本定义，再关联结果表取指标字段。
- 若已确认唯一关联键，是否移除了不必要的二次去重与“取最新”逻辑。
- 是否对“截至当前”口径显式声明并实现了统计锚点时间。

## 输出门槛

- 未完成场景确认，不生成 SQL。
- 多场景未指定主场景，不生成 SQL。
- 关联方向未确认，不生成 SQL。
